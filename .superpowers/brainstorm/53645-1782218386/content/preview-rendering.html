This is the big architectural fork. Your data model is chapters → sections (markdown content), and the server can already export real .docx / .tex. Pick the editing surface — click your preference.

A · Native "paper" cards  Recommended

Pages rendered as native white cards (title page, résumé, TOC, chapters) that look like documents but reflow. Markdown + tables render natively. Tap a section → chat targets it. No true pagination.

✓ Fast, smooth · ✓ Live edits instant · ✓ Matches your data model · ✓ Tables easy

✗ Not pixel-exact to the printed page

— 1 —

B · True A4 pagination (WebView)

A WebView renders real paginated A4 pages with the template's exact fonts/margins (via paged.js). Looks like the printed thesis. Editing happens through chat; re-render after each change.

✓ Pixel-faithful to print · ✓ Exact margins/fonts

✗ Heavy · ✗ Re-paginate lag on edits · ✗ WebView↔native bridge complexity

Page 1 / 24 · 🔒 read-only

C · Server-rendered PDF preview

Server generates the real PDF (reusing the export pipeline); app shows it in a PDF viewer. Editing is a separate native mode; you regenerate the PDF to see the "real" look.

✓ 100% accurate · ✓ Reuses export engine

✗ Edit↔preview split (not live) · ✗ Round-trip latency per change

My recommendation: A as the live working surface — it's the only one where chat-driven edits feel instant, and it fits your chapters/sections model. We can add a "📄 Exact preview" button later that uses C (your export engine already exists) for a true print check. B is the most faithful but fights the live-editing goal.
